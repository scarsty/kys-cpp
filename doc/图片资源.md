# 图片资源

## 资源组织与贴图规则

游戏资源以单张图片的形式放在 `resource/` 的各个目录中，每张图的偏移保存在 `index.ka` 或 `index.txt`。`index.ka` 为连续的 16 位整数对，按图片编号顺序存放；`index.txt` 为文本格式，形式为 `图片编号: 偏移x, 偏移y`，不需要按顺序，两个偏移均为 0 时可以省略。读取时优先使用 `index.txt`，缺失或为空时回退到 `index.ka`。

纹理文件名就是图片编号，不使用前导零：例如编号 0、12 应分别命名为 `0.png`、`12.png`（或统一使用 `.webp`）。初始化时会扫描文件名中出现的第一个数字，以其中的最大值确定纹理组范围；因此应使用纯数字主文件名，避免在无关图片名中加入数字。`index.ka`、`index.txt` 只保存偏移，不能单独创建一个可加载的图片编号。

渲染调用中的“路径”和“编号”分别对应纹理组目录和图片文件名。例如 `renderTexture("head", 17, ...)` 读取 `resource/head/17.png` 或 `resource/head.zip` 内的 `17.png`；`renderTexture("smap", 301, ...)` 对应 `resource/smap/301.png`；`renderTexture("fight/fight017", 3, ...)` 则对应 `resource/fight/fight017/3.png`。`fight017` 中的三位补零属于目录名，是调用方构造的路径，和纹理组内部图片编号的无前导零规则无关。常见的 `title`、`head`、`item`、`mmap`、`smap`、`cloud`、`battle-earth` 和 `*-earth` 都遵循同一“路径/编号”映射。

渲染时会在目标坐标上减去该编号在索引中的偏移，即图片左上角实际绘制于 `(x - 偏移x, y - 偏移y)`。因此偏移用于让素材以统一锚点对齐，不参与文件名查找；替换图片后若锚点改变，也要同步更新该编号的偏移。

若一个编号需要多张动画图，可不提供 `12.png`，而改用 `12_0.png`、`12_1.png`，最多到 `12_9.png`。运行时只有在 `12.png` 不存在时才会按顺序查找这些子纹理；若主文件存在，带下划线的文件不会参与该编号。跨帧绘制时，子纹理按展示帧号对数量取模顺序循环，默认刷新间隔下约为每秒 60 次；同一编号在同一帧内被再次绘制时，后续绘制会随机选择子纹理，以避免多处水面、瀑布等效果完全同步。

纹理组支持 PNG 与 WebP。一个纹理组只能使用一种图片扩展名，不能在同一目录或 ZIP 中混用 PNG 与 WebP。默认先读取与目录同名的 ZIP，例如 `resource/head.zip`；若 ZIP 不存在则读取 `resource/head/` 目录。`png_from_path=1` 可强制只读目录，`load_all_png=1` 会在纹理组初始化时预加载全部图片，默认按需加载。

战斗贴图中，每个人的动作帧数用 `fightframe.txt` 保存。格式为动作索引（0~4）及每方向数量；未写的动作视为 0。该文件通过纹理组接口读取，因此可与战斗图片一起放在目录或同名 ZIP 中。

### 图片偏移补充

也可以将偏移保存在 PNG 文件中，但这是可选信息，当前资源不采用此方式。`abc/png_offset.h` 和 `abc/png_offset.cpp` 提供了修改 PNG 偏移的例程。

## 高清素材

### 主地图

游戏中需要将碎图拼成一张完整的图片。如使用算法或深度学习的模型将素材放大，直接拼接的话会出现接缝。因此，应先拼接成一张完整的地面图片（UpEdit 有此功能），再将其放大。之后在游戏中再往上画建筑等元素。

整张地面图使用 RealESRGAN 的“3D写实”类别和 `Photo-HQ-W4xEX` 模型放大。

因为主地图的尺寸较大，已经超过显卡的纹理尺寸，因此应使用程序拆散为一系列图片。此处是 8×8 的方案，生成 64 张图片。可以手动删除一些没有内容的图片以节省空间。

一个范例程序如下：

```c++
int main()
{
    cv::Mat m = cv::imread("map.bmp");
    autocrop(m);
    size_t size = 8;
    int width = m.cols / size;
    int height = m.rows / size;
    int k = 0;
    for (int i = 0; i < size; i++)
    {
        for (int j = 0; j < size; j++)
        {
            cv::Mat m1 = m(cv::Rect(j * width, i * height, width, height));
            double minv, maxv;
            cv::minMaxLoc(m1, &minv, &maxv);
            if (minv != maxv)
            {
                cv::imwrite(std::format("{}.png", k), m1);
                std::print("{}: {},{}\n", k, minv, maxv);
            }
            k++;
        }
    }
    return 0;
}
```

之后将生成的图片放入 `mmap-earth`，即可画出地面。这种方式不支持地面图片闪烁，但可以用其他方法增加动态。

其他素材都是独立的图片，直接放大即可。

#### 单图放大模型

单独的角色、头像、物品、特效和战斗帧使用 RealESRGAN，模型类别选择“3D写实”，模型选择 `RealESRGAN-x4plus`。该模型的原始输出为 4 倍；项目目标若是 2 倍资源，必须在放大后再等比例缩小到 2 倍，或使用工具中等价的 2 倍输出选项。不要将模型名称中的 `x4` 直接当作资源的最终缩放倍数。

放大模型不是项目规则。选用何种模型属于个人审美和素材效果取舍，可以根据图片类型、输出效果与人工检查结果更换模型。无论使用何种模型，后续的最终尺寸、偏移、帧编号和资源格式约束仍然适用。

最终图片的宽和高必须相对于原图使用同一个整数比例。后续偏移只按最终尺寸相对于原图的比例计算：例如模型先放大到 4 倍、再缩小为 2 倍时，偏移仍只乘 2，而不是乘 4。

### 内景地图和战斗地图

预先拼好地面再放大放置于 `smap-earth` 和 `battle-earth`，素材再单独放大。内景地图的整张地面图同样使用 RealESRGAN 的“3D写实 / `Photo-HQ-W4xEX`”模型。但是有一些墙体和占用多张图的物体仍然有接缝，这个问题暂时不好解决。

### 其他素材

光影和战斗图单独放大即可。战斗目录中的图片编号和数量不能改变：例如原目录含 `0` 到 `43` 共 44 帧，放大后仍应保留连续的 `0` 到 `43`。`fightframe.txt` 记录动作帧布局，必须与最终图片数量保持兼容。

### 偏移的设置

因为放大后图片的尺寸发生了变化，所以需要重新设置偏移。一般来说直接乘以最终放大倍数即可。可以使用 `tools/scale_index_ka.py`：

```shell
python tools/scale_index_ka.py resource/fight/fight211 --factor 2
```

脚本会同时处理 `index.ka` 和 `index.txt`；目录会递归处理，传入 ZIP 时会原地重写 ZIP 内根目录的索引文件。`index.ka` 是按图片编号排列的 little-endian 有符号 16 位整数对 `(dx, dy)`，因此放大后两个坐标都必须乘以相同倍数，并且结果不能超出 $[-32768, 32767]$。`index.txt` 与 `index.ka` 同时存在时运行时优先读文本索引，所以两份索引必须同步缩放并逐项一致。

已经缩放过的资源不能再次对父目录批量运行脚本，否则偏移会再次乘倍。处理前应限定到尚未缩放的单个目录，或建立独立的待处理目录。

#### 图片格式与压缩

转换一整个战斗目录时，应在所有 PNG 都成功转换后再删除 PNG，保证该组最终只剩 WebP。

Dragon 的 WebP 使用有损 `VP8 ` 编码。新增 Dragon 高清单图使用 `--lossy --quality 80`：

```shell
python tools/convert_png_to_webp.py <待处理目录> --lossy --quality 80
```

该脚本会原地将 PNG 替换为 WebP；处理 ZIP 时会原地重打包。因此必须先备份或只对独立的待发布目录操作。

转换后至少检查：目录中不再残留 PNG；WebP 图片编号连续；图片数量与 `index.ka` 的偏移对数一致；并确认索引文件和 `fightframe.txt` 仍保留。

### 程序中的修改

`TILE_W` 和 `TILE_H` 的值需要修改为放大后的尺寸。其他还有地面高度、装饰高度等。

这里使用 `Scene::setTileScale` 设置放大倍数，可以在 INI 中设置 `tile_scale` 参数控制。

### 内场景的画图逻辑

根据反编译结果，应该是先画出无高度地面（也可能是高度小于某值），之后其他的地面和建筑层一起画出。

而在高清版中，因为地面需要预先放大拼合后的图片，为了避免出现裂缝，要将所有地面一起画出。

## Android 资源发布

Android 发布包将完整游戏资源打为 `assets/game.zip`。首次启动时，程序会将其解压到 `/sdcard/kys-cpp/game/` 并写入完成标记；之后从该可写游戏目录读取资源和存档。
