
# 资源、存档与 abc 工程

存档的基础数据部分（R 部分）使用 SQLite 数据库保存。默认启用 `zip_save` 时，每个存档槽保存为 `save/<编号>.zip`：其中包含 `1.db`，以及未启用数据库 SD 数据时的 `s1.grp`、`d1.grp`。关闭 `zip_save` 时，R、S、D 分别保存为 `<编号>.db`、`allsin.grp`/`s<编号>.grp`、`alldef.grp`/`d<编号>.grp`。

读取存档时会优先读取 ZIP；若 ZIP 不存在，仍可回退到未压缩的 R/S/D 文件，并在启用 `zip_save` 时转换为 ZIP 存档。S、D 数据也可选择写入 SQLite 的 BLOB 表，但该选项默认关闭。

之前游戏使用的列表文件只保留了升级经验列表和离队列表，改用txt格式。

DOS 版初始存档中的中文字段可能仍是 CP950（BIG5）。abc 的扩展转换会将这些字符串转为 UTF-8，并导出 SQLite 数据库；`BaseInfo::Encode` 字段会保留在数据库中作为兼容数据。当前运行时的 `Save` 路径不根据该字段做文本编码转换，因此不要依赖“保存时自动转换编码”的旧行为。

abc工程用来转换之前的数据。建议自行调整代码后，使用调试模式执行。

其中主要的功能是将 DOS 版的 16 位 GRP/IDX 存档扩展为 32 位数据，转换 CP950 字符串，并将 R 部分导出为 SQLite 数据库。

存档的S和D部分仍使用16位整数，其余的还有一些其他文件也没有进行转换。

以上提到的数据，除了文本文件外均可以用真正的强强的新版upedit修改（该修改器不完善）。

游戏资源以单张图片的形式放在 `resource/` 的各个目录中，每张图的偏移保存在 `index.ka` 或 `index.txt`。`index.ka` 为连续的 16 位整数对，按图片编号顺序存放；`index.txt` 为文本格式，形式为 `图片编号: 偏移x, 偏移y`，不需要按顺序，两个偏移均为 0 时可以省略。读取时优先使用 `index.txt`，缺失或为空时回退到 `index.ka`。

纹理文件名就是图片编号，不使用前导零：例如编号 0、12 应分别命名为 `0.png`、`12.png`（或统一使用 `.webp`）。初始化时会扫描文件名中出现的第一个数字，以其中的最大值确定纹理组范围；因此应使用纯数字主文件名，避免在无关图片名中加入数字。`index.ka`、`index.txt` 只保存偏移，不能单独创建一个可加载的图片编号。

渲染调用中的“路径”和“编号”分别对应纹理组目录和图片文件名。例如 `renderTexture("head", 17, ...)` 读取 `resource/head/17.png` 或 `resource/head.zip` 内的 `17.png`；`renderTexture("smap", 301, ...)` 对应 `resource/smap/301.png`；`renderTexture("fight/fight017", 3, ...)` 则对应 `resource/fight/fight017/3.png`。`fight017` 中的三位补零属于目录名，是调用方构造的路径，和纹理组内部图片编号的无前导零规则无关。常见的 `title`、`head`、`item`、`mmap`、`smap`、`cloud`、`battle-earth` 和 `*-earth` 都遵循同一“路径/编号”映射。

渲染时会在目标坐标上减去该编号在索引中的偏移，即图片左上角实际绘制于 `(x - 偏移x, y - 偏移y)`。因此偏移用于让素材以统一锚点对齐，不参与文件名查找；替换图片后若锚点改变，也要同步更新该编号的偏移。

若一个编号需要多张动画图，可不提供 `12.png`，而改用 `12_0.png`、`12_1.png`，最多到 `12_9.png`。运行时只有在 `12.png` 不存在时才会按顺序查找这些子纹理；若主文件存在，带下划线的文件不会参与该编号。跨帧绘制时，子纹理按展示帧号对数量取模顺序循环，默认刷新间隔下约为每秒 60 次；同一编号在同一帧内被再次绘制时，后续绘制会随机选择子纹理，以避免多处水面、瀑布等效果完全同步。一个纹理组应统一使用 PNG 或 WebP，不能混用两种扩展名。

纹理组支持 PNG 与 WebP。默认先读取与目录同名的 ZIP，例如 `resource/head.zip`；若 ZIP 不存在则读取 `resource/head/` 目录。`png_from_path=1` 可强制只读目录，`load_all_png=1` 会在纹理组初始化时预加载全部图片，默认按需加载。

战斗贴图中，每个人的动作帧数用 `fightframe.txt` 保存。格式为动作索引（0~4）及每方向数量；未写的动作视为 0。该文件通过纹理组接口读取，因此可与战斗图片一起放在目录或同名 ZIP 中。

Android 发布包将完整游戏资源打为 `assets/game.zip`。首次启动时，程序会将其解压到 `/sdcard/kys-cpp/game/` 并写入完成标记；之后从该可写游戏目录读取资源和存档。

## 关于图片偏移的补充

也有一个思路是将偏移保存在png文件之中，需注意png文件本身是可以包含偏移的，但是一般情况是可选项。该思路基本验证通过，但是目前并没有使用。

在abc文件中有png_offset.h和png_offset.cpp文件，其中提供了修改png文件偏移的例程。libpng这个库的API接口有点古老，因此程序看起来比较繁琐。
