# Magic Effect System Design

## Goal

Add configurable identity effects for every magic skill used by `chess_pool`, while making magic effects share the same trigger, chance, count, timer, and application flow as the existing combo effects.

The key design principle is to avoid creating two parallel systems. A magic effect is another effect source, not another effect engine.

## Problem Recap

The rejected implementation attempted to store `std::vector<ComboEffect>` in `BattleActionSkillSeed`, `BattleCastSkillState`, `BattleAttackPayload`, `BattleAttackEvent`, and `BattleHitSkillSnapshot`, then manually merge combo effects and magic effects inside hit resolution.

That shape has three problems:

- Performance: attack events are frequent, and `Moved` events are produced every frame. Carrying effect vectors in `BattleAttackPayload` / `BattleAttackEvent` causes unnecessary copying.
- Design: the hit resolver has to duplicate logic in multiple places: once for combo effects, once for magic effects.
- Correctness: `RoleComboEffectId` is local to one effect state. If combo and skill stores are merged without source identity, activation counts, max-count limits, pending flags, timers, counters, stacks, and logs can update the wrong state.

## Core Direction

Keep the mature indexing and runtime-state model currently inside `RoleComboState`, but abstract it into a generic effect state.

Suggested shape:

```cpp
class BattleEffectState
{
public:
    RoleComboEffectId applyConfiguredEffect(const ComboEffect& effect, int sourceId);
    const RoleComboEffectInstance& effect(RoleComboEffectId id) const;
    std::span<const RoleComboEffectId> effectIds(Trigger trigger, EffectType type) const;
    std::span<const RoleComboEffectId> effectIdsInAppendOrder() const;

    bool canActivateTriggeredEffect(RoleComboEffectId id) const;
    void recordTriggeredEffectActivation(RoleComboEffectId id);
    int triggeredEffectActivationCount(RoleComboEffectId id) const;

    void setTypePending(EffectType type, bool value);
    bool consumeTypePending(EffectType type);
    bool advanceEffectCounter(RoleComboEffectId id, int threshold);
    void setEffectFrameTimer(RoleComboEffectId id, int frames);
    bool advanceEffectFrameTimer(RoleComboEffectId id, int intervalFrames);
    RoleComboStackChange recordEffectStack(RoleComboEffectId id, int maxStacks, int pctPerStack);
    RoleComboStackChange recordEffectStackAgainst(RoleComboEffectId id, int unitId, int maxStacks, int pctPerStack);

    int sumAlways(EffectType type) const;
    int maxAlways(EffectType type) const;
    int maxAlwaysValue2(EffectType type) const;
    const RoleComboEffectInstance* firstAlways(EffectType type) const;
    bool hasAlways(EffectType type) const;
};

class RoleComboState : public BattleEffectState
{
};
```

`RoleComboState` can remain as the combo-facing name, so existing combo callers can migrate gradually. The reusable indexing and mutable runtime behavior should live in `BattleEffectState`.

## Definition And Runtime Ownership

Separate immutable parsed definitions from mutable per-unit runtime state.

Immutable definitions:

- Loaded once from `config/chess_magic_effects.yaml`.
- Indexed by magic ID.
- Contain parsed `ComboEffect` definitions only.
- Do not contain activation counts, pending flags, counters, timers, or stacks.

Mutable runtime state:

- Lives on each `BattleRuntimeUnitRecord`, next to `combo`.
- Contains one `BattleEffectState` for the normal skill slot and one for the ultimate skill slot.
- Is built during battle initialization by copying configured effects from the immutable definition store into the unit's normal/ultimate slot states.
- Persists for the whole fight, not per cast.

The important rule is that action and cast structs must not own mutable effect state. `BattleActionSkillSeed`, `BattleCastSkillState`, and `BattlePendingCastAction` are value-copied today; if they carry mutable effect stores, counters and pending flags will fork. They may carry magic IDs and lightweight refs, but all mutation must resolve back to the owning `BattleRuntimeUnitRecord`.

Suggested runtime shape:

```cpp
struct BattleUnitSkillEffectState
{
    int magicId = -1;
    BattleEffectState effects;
};

struct BattleUnitMagicEffectRuntime
{
    BattleUnitSkillEffectState normal;
    BattleUnitSkillEffectState ultimate;
};
```

Then add `BattleUnitMagicEffectRuntime skillEffects;` to `BattleRuntimeUnitRecord`.

Clone rule:

- Clones rebuild fresh `skillEffects` from their copied `BattleActionPlanSeed`.
- Clones inherit the copied action plan's normal/ultimate magic IDs, but do not inherit skill-effect activation counts, pending flags, counters, timers, or stacks from the source unit.
- If a clone has no action plan, both skill effect slots are empty.
- `CloneSummon` remains invalid as a magic effect in phase one, so magic effects cannot recursively define clone creation behavior.

## Source-Aware Effect Identity

`RoleComboEffectId` stays local to a single `BattleEffectState`. Any API that crosses a source boundary must use a source-aware ref.

Suggested shape:

```cpp
enum class BattleEffectSourceKind
{
    Combo,
    Skill,
};

enum class BattleSkillSlot
{
    None,
    Normal,
    Ultimate,
};

struct BattleEffectStoreRef
{
    BattleEffectSourceKind kind = BattleEffectSourceKind::Combo;
    BattleSkillSlot slot = BattleSkillSlot::None;
};

struct BattleEffectRef
{
    BattleEffectStoreRef store;
    RoleComboEffectId localId;
};

struct BattleEffectSource
{
    BattleEffectStoreRef ref;
    BattleEffectState* state = nullptr;
};

struct BattleEffectSources
{
    BattleEffectSource combo;
    BattleEffectSource skill;
};

struct BattleEffectTriggerEvent
{
    BattleComboTriggerHook hook{};
    int sourceUnitId{};
    int targetUnitId{};
    BattleEffectRef effectRef;
    ComboEffectSnapshot effect;
};
```

Rules:

- Inside one `BattleEffectState`, keep using `RoleComboEffectId`.
- Once results may contain both combo and skill effects, return `BattleEffectRef` or `BattleEffectTriggerEvent`, not plain `BattleComboTriggerEvent`.
- Caller-recorded activation must go through a source-aware mutation helper such as `BattleEffectCommands::recordActivation(sources, effectRef)`.
- Logs and test assertions that report activated effects should include the source kind and skill slot, not only the local ID.

## Skill Effect References In Attack Events

Do not copy effect contents into attack events. Attack payloads/events should carry only enough information to resolve the selected skill effect state from the runtime unit record.

Suggested shape:

```cpp
struct BattleSkillEffectRef
{
    int unitId = -1;
    BattleSkillSlot slot = BattleSkillSlot::None;
};
```

`BattleAttackPayload` and `BattleAttackEvent` should carry `BattleSkillEffectRef`, plus the existing `ultimate` and `mainProjectile` flags. Hit resolution resolves the ref through `BattleRuntimeUnits`:

- `slot == Normal` resolves to `state.units.require(unitId).skillEffects.normal.effects`.
- `slot == Ultimate` resolves to `state.units.require(unitId).skillEffects.ultimate.effects`.
- `slot == None` means no skill effect source.

This keeps event payloads small and avoids copying mutable state through spawn, move, hit, block, bounce, and expiration events.

## Shared Query And Mutation API

Add a read/query layer that combines sources but never hides source identity.

```cpp
class BattleEffectReader
{
public:
    int sumAlways(const BattleEffectSources& sources, EffectType type) const;
    int maxAlways(const BattleEffectSources& sources, EffectType type) const;
    int maxAlwaysValue2(const BattleEffectSources& sources, EffectType type) const;
    bool hasAlways(const BattleEffectSources& sources, EffectType type) const;
    const RoleComboEffectInstance* firstAlways(const BattleEffectSources& sources, EffectType type) const;
    BattleEffectRef firstAlwaysRef(const BattleEffectSources& sources, EffectType type) const;

    std::optional<ComboEffectSnapshot> maxAlwaysPair(
        const BattleEffectSources& sources,
        EffectType type) const;

    std::vector<BattleEffectTriggerEvent> collectTriggerEvents(
        const BattleEffectSources& sources,
        const BattleComboTriggerInput& input,
        std::initializer_list<EffectType> effectTypes,
        BattleRuntimeRandom& random,
        BattleComboActivationRecording recording) const;
};
```

Put source-aware mutations in a separate command/helper type. These methods are intentionally not `const`, even if the helper itself has no fields, because they mutate the referenced effect states.

```cpp
class BattleEffectCommands
{
public:
    void recordActivation(const BattleEffectSources& sources, BattleEffectRef effectRef);
    bool consumeTypePending(const BattleEffectSources& sources, EffectType type);
    bool advanceEffectCounter(const BattleEffectSources& sources, BattleEffectRef effectRef, int threshold);
};
```

Mutation rules:

- Source scans such as `consumeTypePending` skip null sources.
- A `BattleEffectRef` that names a missing source or a local ID that does not belong to that source is an invariant violation and should assert.
- Caller-recorded trigger events must be recorded through `BattleEffectCommands`, not by reaching into `RoleComboState` directly.

Merge rules:

- `sumAlways`: add values from all active sources.
- `maxAlways`: maximum `value` across all active sources.
- `maxAlwaysValue2`: maximum `value2` across all active sources only when the caller truly wants independent `value2` aggregation.
- Paired `value` / `value2` effects must use `maxAlwaysPair`, `firstAlways`, or source-aware iteration so `value2` comes from the same effect instance as `value`.
- `hasAlways`: logical OR across all active sources.
- `firstAlways`: deterministic source order, combo first and skill second. Use this only for effects whose semantics are explicitly "first matching effect wins".
- Trigger collection: collect from combo first, then skill; preserve each state's local sorted order; return source-aware events.
- Pending flags: consuming a pending type should consume it from each active source and return true if any source had it pending.
- Counters, timers, stacks, and activation counts: mutate only through `BattleEffectRef`, never through a bare local ID.

Default stacking rule for phase one: skill effects stack with combo effects according to the accessor merge rules above. There is no override/replace behavior unless a later YAML field explicitly introduces it.

## `Trigger::Always` Scope

For combo effects, `Trigger::Always` means unit-passive. It participates in battle start, frame runtime, status state, damage state, movement state, cast planning, and hit resolution.

For magic effects, `Trigger::Always` means selected-skill-scoped. It is active only while resolving the selected skill's cast, spawned projectiles, and hits. It must not become a passive unit aura.

Scope matrix for magic effects:

| Area | Magic `Always` rule |
| --- | --- |
| Battle-start state such as `FlatShield`, `ShieldPctMaxHP`, `BlockFirstHits`, `DeathPrevention`, `KillHealPct`, `KillInvincFrames`, `Bloodlust`, `CloneSummon` | Invalid for magic effects in phase one. These are unit-passive combo effects. |
| Frame runtime such as low-HP, aura, auto-ultimate timers, heal aura, HP regen, rescue, last-alive effects | Invalid for magic effects in phase one. There is no selected skill during frame tick. |
| Cast planning and movement identity such as `ForceRangedAttack`, `DashAttack`, `BlinkAttack`, movement dash behavior | Invalid for magic effects in phase one. These influence whether and how a unit chooses an action before a skill slot is resolved. |
| Selected-cast effects such as `PostSkillInvincFrames`, `UltimateExtraProjectiles`, cast-scoped shield grants, global cast blasts, team MP restore | Valid only for the selected skill slot at cast commit / cast finish. |
| Cooldown effects such as `CDR` | Valid only as selected-skill cooldown shaping after a cast is selected. It must not affect unrelated casts or action planning. |
| Offensive hit effects such as damage modifiers, armor penetration, on-hit MP/HP gain, poison, stun, damage-reduction debuff, MP block, nearby follow-up projectiles | Valid only for hit events whose `BattleSkillEffectRef` resolves to that skill state. |
| Defensive hit effects such as projectile reflect, skill reflect, on-being-hit stun, shield-break triggers | Invalid for magic effects in phase one. A defender does not have a selected attacking skill. |

Loader validation should reject a magic effect if its trigger/type pair has no defined selected-skill scope. Do not silently load inert effects.

## Selected-Skill Cooldown Integration

Current cast code fills `BattleCastInput::unit.cooldownReductionPct` from combo state before cooldown calculation, and `cooldownForOperation` consumes that field. Selected-skill `CDR` must not become a passive unit field.

Preferred phase-one integration:

- Select the skill first.
- Resolve the selected skill slot into `BattleEffectSources { combo, selectedSkill }`.
- Compute `BattleCastInput::unit.cooldownReductionPct` as the combined selected-cast CDR for this one cast.
- Call `cooldownForOperation` with that already-combined value.
- For pending casts, recompute the combined selected-cast CDR from the pending skill ref at commit time instead of trusting a copied value from `BattlePendingCastAction`.

A later cleanup may pass `BattleEffectSources` or an explicit `selectedCastCooldownReductionPct` into `cooldownForOperation`, but phase one should avoid a wide `BattleCastSystem` signature churn unless it is already touching that file.

## Paired Runtime Effects

Some existing runtime effects are intentionally split across effect types. The magic loader must use the same representation rather than adding a magic-only path.

`攻击冷却延长` currently parses as `EffectType::OffensiveCharm`; runtime then reads chance from `OffensiveCharm` and cooldown extension percent from a paired `CharmCDRDebuff` effect. For magic YAML, the loader should normalize:

```yaml
- 类型: 攻击冷却延长
  数值: 30
  附加参数: 35
  触发: 攻击命中时概率
  触发参数: 100
```

into the same shared runtime shape:

- `OffensiveCharm`: chance `30`.
- `CharmCDRDebuff`: chance `30`, cooldown extension percent `35`.
- Both effects use the same trigger, trigger parameter, duration, and max-count fields from the YAML entry.

Hit resolution should then use the normal paired-effect lookup from `BattleEffectReader`; it should not add a separate magic-only cooldown-extension branch.

## Trigger Semantics

`AfterSkillCast` currently maps to both `OnCast` and `OnUltimate`. That must become input-aware.

Suggested extension:

```cpp
struct BattleComboTriggerInput
{
    BattleComboTriggerHook hook = BattleComboTriggerHook::FrameTick;
    int sourceUnitId = -1;
    int targetUnitId = -1;
    bool ultimate = false;
    bool mainProjectile = true;
};
```

Rules:

- `攻击出手时概率`: can trigger for both normal skills and ultimates.
- `绝招时`: can trigger only when `ultimate == true`.
- `攻击命中时概率`: can use `mainProjectile` to decide whether side projectiles, splash hits, bounce hits, or follow-up projectiles may trigger the effect.

Implementation plumbing must touch these places:

- `BattleComboTriggerSystem::triggersForHook` should become input-aware, or filtering should happen immediately after it returns candidate triggers. `OnUltimate` must be filtered out unless `input.ultimate` is true.
- `BattleCore.cpp` call sites that build `BattleComboTriggerInput{ BattleComboTriggerHook::AfterSkillCast, ... }` must pass the actual cast `ultimate` flag. This includes pending cast commit, skill finish team-heal collection, extra projectile collection, and cast-scoped combo effect collection.
- `BattleAttackPayload` already has `ultimate` and `mainProjectile`; `BattleAttackEvent` already copies those fields. Hit-resolution call sites that build `DamageDealt`, `ProjectileHitEnemy`, or related trigger inputs must pass `input.attackEvent.ultimate` and `input.attackEvent.mainProjectile`.
- Shield-break and damage-taken trigger inputs should pass `ultimate = false` and `mainProjectile = true` unless a future feature has a real selected skill source for those hooks.

## YAML Loading And Validation

Add `config/chess_magic_effects.yaml`. The format should reuse the existing combo-effect fields:

```yaml
武功效果:
  - 武功: 26
    名称: 降龍十八掌
    用途: 絕招
    效果:
      - 类型: 眩晕
        数值: 14
        触发: 攻击命中时概率
        触发参数: 45
```

`用途` is review metadata only. It should not directly control mechanics. Real triggering still comes from `触发`, and real ultimate behavior comes from the selected skill slot plus `BattleComboTriggerInput::ultimate`.

Loader validation should require:

- The `武功效果` root node exists.
- `武功` IDs are unique.
- Each magic has at least one `效果`.
- Every effect parses through `ChessBattleEffects::parseEffect`.
- Every parsed trigger/type pair has a defined magic scope from the matrix above.
- If an `EffectType` has no runtime application path, loading fails with a clear message instead of silently doing nothing.

## New Effect: Lowest Ally Heal

`最低友方治療` can remain, but it should enter the same shared effect flow as a normal `EffectType`.

Value semantics:

- `数值` is flat HP healed.
- `附加参数` is optional percent of the target's max HP healed.
- Total heal is `数值 + target.maxHp * 附加参数 / 100`.
- Source can heal self if self is the lowest valid ally.
- Select only living same-team units that are not already at full HP.
- Select the unit with the lowest current HP percentage.
- Break ties by lower unit ID for deterministic behavior.
- Emit no gameplay event, log event, or visual event if there is no valid target.
- Emit a heal command/log using reason `最低友方治療` unless the caller supplies a more specific skill reason.

This effect should currently be reserved for real doctor skills, such as `青囊奇術`.

## Suggested Implementation Order

1. Keep the rejected implementation stashed; do not restore the parts that copy `std::vector<ComboEffect>` into attack events.
2. Extract `BattleEffectState`, with `RoleComboState` becoming a thin wrapper or compatibility subclass.
3. Add source-aware refs and `BattleEffectSources`.
4. Add `BattleEffectReader` with the complete read accessor set: `sumAlways`, `maxAlways`, `maxAlwaysValue2`, `hasAlways`, `firstAlways`, paired lookup, and trigger collection.
5. Add `BattleEffectCommands` for activation recording, pending consumption, and counter/timer mutation.
6. Add unit-owned `BattleUnitMagicEffectRuntime` and build normal/ultimate skill states during battle initialization from the immutable magic definition store.
7. Make clone creation rebuild fresh skill effect states from the copied action plan.
8. Add `BattleSkillEffectRef` to attack payloads/events and pending/cast paths only as lightweight identity.
9. Make trigger plumbing pass `ultimate` and `mainProjectile` through the call sites listed above.
10. Migrate hit resolver and cast-scoped application to use `BattleEffectSources` while keeping one application path per effect type.
11. Add selected-skill CDR integration after skill selection.
12. Add magic YAML loading and validation, including trigger/type scope validation and paired runtime-effect normalization.
13. Add `最低友方治療` through the shared effect path.
14. Wire in `config/chess_magic_effects.yaml`.

## Test Focus

- YAML loads successfully, and duplicate magic IDs fail.
- YAML fails when a magic effect uses a trigger/type pair that has no selected-skill scope.
- Skill runtime state persists per unit skill slot for the whole fight; copied action/cast structs do not fork activation counters or pending flags.
- Clones rebuild fresh skill effect states from copied action plans and do not inherit the source unit's skill-effect counters, pending flags, timers, or stacks.
- The same unit can have different magic effect states for normal and ultimate skills.
- `BattleEffectReader` can see both combo effects and selected-skill effects.
- Combo and skill local effect IDs can collide without corrupting activation counts, counters, timers, pending flags, or logs.
- One shared hit flow can trigger combo stun and magic stun without duplicate application logic.
- `OnUltimate` does not trigger for normal skill casts.
- Side projectiles do not trigger effects that are restricted to the main projectile.
- Paired value/value2 effects keep both values from the selected effect instance.
- `CDR` from a magic effect affects only the selected cast cooldown.
- `攻击冷却延长` normalizes to the same `OffensiveCharm` plus `CharmCDRDebuff` runtime pair used by combo effects.
- Unit-passive effects such as `FlatShield`, `DeathPrevention`, and `ForceRangedAttack` are rejected for magic effects unless a future phase gives them explicit selected-skill semantics.
- `最低友方治療` selects the living same-team unit with the lowest HP percentage, can choose the source, and uses flat plus optional target-max-HP percent healing.
