# 武功效果路徑整併建議

## 背景

目前武功效果已經做到幾件重要的事：

- `config/chess_magic_effects.yaml` 會被載入並轉成和羈絆效果相同的 `ComboEffect`。
- 每個單位有自己的 `skillEffects` runtime state，避免把 mutable effect state 複製進 action、cast、attack event。
- `BattleEffectSources` 可以同時表示羈絆來源與當前選中武功來源。
- `BattleEffectReader` 已經能對多來源做 `sumAlways`、trigger collection、paired value/value2 查詢等操作。
- 棋子面板可以顯示武功效果描述。

但最近的毒效果問題暴露出一個還沒有真正收斂的地方：資料與顯示已經共用，但實際 runtime 應用仍散落在多個手寫分支。`中毒伤害` 在 YAML 中是 `攻击命中时概率`，面板能顯示，loader 也接受，但 hit resolver 當時只讀 `PoisonDOT` 的 `Always` summary，所以 selected-skill 的 OnHit 毒效果不會生效。

這不是單一毒效果的問題，而是系統還處在「半整併」狀態。

## 已定案方向

- Runtime capability registry 放在新的 battle-side 檔案，例如 `src/battle/BattleEffectRuntimeRegistry.h/.cpp`。
- Registry 必須成為 selected-skill scope 與 runtime phase 的唯一權威來源；loader 不能再保留一份獨立的 accepted pair switch。
- 真實 `config/chess_magic_effects.yaml` 不作為 unit test fixture。
- 不新增專門的外部掃描工具。
- 在遊戲啟動或 battle runtime setup 載入武功效果時做驗證。驗證必須便宜，只查靜態表，不做昂貴掃描。
- 整併範圍要覆蓋全部目前支援的武功效果路徑，不只 hit effects。Hit、cast commit、cast finish、cooldown、projectile follow-up、armor penetration、frame/battle-start combo adapters 都要有清楚歸屬。

### 目前啟用策略

初期平衡策略是：`chess_magic_effects.yaml` 中的武功效果全部視為絕招效果，只套用在 ultimate skill，不套用在普通技能。

目前 runtime 建立路徑也應維持這個策略：battle setup 只為 `actionPlan->ultimateSkill` 建立 `skillEffects.ultimate`，普通技能槽不帶 selected-skill magic state。也就是說，YAML 裡的效果描述可以使用 `Always`、`OnHit`、`OnCast` 等 trigger 表達「這個絕招在戰鬥中如何觸發」，但它們的來源仍只會掛在絕招上。

`用途:` 目前只作為顯示或分類 metadata，不控制 runtime 行為，也不應被拿來判斷普通技能或絕招技能。

## 目標

降低未來新增或調整武功效果時漏接 runtime 路徑的機率。

核心目標有三個：

- 一個 magic effect 若能被 loader 接受，就必須有已登記的 runtime phase。
- 羈絆效果與武功效果不應各自擁有一套 runtime 實作。
- Runtime 應以「戰鬥階段」整併，而不是以「來源種類」整併。

正確形狀應該是：

```text
combo effects + selected ultimate skill effects
    -> BattleEffectSources
    -> phase resolver
    -> gameplay commands / damage modifiers / logs
```

而不是：

```text
combo path
skill path
special magic path
```

## 設計

### BattleEffectRuntimeRegistry

新增 battle-side registry，描述 engine 對 `EffectType + Trigger` 的 runtime 支援能力。

建議檔案：

- `src/battle/BattleEffectRuntimeRegistry.h`
- `src/battle/BattleEffectRuntimeRegistry.cpp`

建議資料形狀：

```cpp
enum class BattleEffectRuntimePhase
{
    BattleStart,
    FrameRuntime,
    CastPlanning,
    CastCommit,
    CastFinish,
    HitPreDamage,
    HitDamageModifier,
    HitStatusPayload,
    HitResourcePayload,
    HitFollowUp,
};

struct BattleEffectRuntimeRule
{
    EffectType type;
    Trigger trigger;
    BattleEffectRuntimePhase phase;
    bool selectedSkillMagicAllowed;
    bool ultimateOnly;
};
```

Registry 應提供便宜查詢：

```cpp
bool isSelectedSkillMagicAllowed(EffectType type, Trigger trigger);
BattleEffectRuntimePhase runtimePhaseFor(EffectType type, Trigger trigger);
bool isUltimateOnlyMagicEffect(EffectType type, Trigger trigger);
```

實作上可用 `static constexpr std::array<BattleEffectRuntimeRule, N>` 加線性搜尋。資料量很小，而且只在載入設定與少量驗證中使用，不需要複雜索引。

重點是：這張 registry 取代 `ChessBattleEffects.cpp` 裡的 `magicEffectHasSelectedSkillScope`。Loader validation 只能問 registry，不能再有另一份 selected-skill accepted list。這樣 registry 才不是新的 drift 來源。

### Cast Finish Magic

Cast-finish magic 加入當前 scope，但只支援 `OnUltimate`。語意是「這個絕招進入 cooldown，且 cooldown 結束時觸發」，不是普通技能完成，也不是 cast commit 當下觸發。Combo 的 `Always` 群療 payload 仍然存在：它在 pending 被消耗後加總，和 triggered `OnUltimate` 群療由同一個 CastFinish resolver 輸出。

這條路徑不能只靠 `cooldownFromUltimate` bool。需要在 runtime unit 或 cooldown state 上保存 source-aware finish ref，至少能回答：

- 這次進 cooldown 的技能是否來自 ultimate。
- 這次 finish 應該使用哪個 selected-skill effect state。
- finish-time trigger activation 應該寫回哪個 source。
- follow-up projectile 或 heal command 是否需要攜帶 `skillEffectRef`。

建議狀態形狀：

```cpp
struct BattleSkillCooldownSource
{
    BattleSkillEffectRef skillEffectRef;
    bool ultimate;
};
```

或直接在 action/cooldown state 中保存 `BattleSkillEffectRef cooldownFinishSkillEffectRef`，並以 `slot == BattleSkillSlot::Ultimate` 判定是否為絕招。因為目前 release policy 是 ultimate-only，普通技能 cooldown 可以保留 empty ref。

finish-time flow 應該是：

```text
CastCommit
    -> mark CastFinish pending on combo + selected ultimate skill sources
    -> store cooldownFinishSkillEffectRef
Cooldown reaches 0
    -> make BattleEffectSources(combo + stored skill ref)
    -> resolve CastFinish once
    -> consume pending state from all participating sources
    -> add combo Always CastFinish payloads from consumed combo source
    -> record triggered activation on the source that owns each effect
    -> clear cooldown finish source
```

這樣羈絆與武功共用同一條 CastFinish resolver，不需要 combo-only 與 magic-only 兩套 heal 邏輯。

### 載入時驗證

Magic YAML validation 應在 parse/normalize 後只做一件事：查 battle-side registry。

Validation 應確認：

- `EffectType + Trigger` 允許作為 selected-skill magic effect。
- `EffectType + Trigger` 有 runtime phase。
- 目前 release policy 要求 magic effect 為 ultimate-only。

任何缺失都應讓載入失敗並印出明確錯誤。

這個驗證發生在遊戲啟動或 battle runtime setup 載入武功效果時。它只查靜態 registry，不讀額外檔案、不跑戰鬥模擬、不掃描 source，所以成本可忽略。

這樣可把錯誤從：

```text
面板看得到效果，但進戰鬥才發現沒有 runtime 路徑
```

提前成：

```text
啟動或載入設定時明確失敗
```

### Phase Resolver

Runtime application 應以 phase resolver 收斂。每個 phase helper 接收 `BattleEffectSources`，不直接接收 `RoleComboState`。

建議拆分：

```cpp
class BattleHitEffectResolver
{
public:
    BattleHitPreDamageEffects resolvePreDamage(
        const BattleEffectSources& attackerSources,
        const BattleHitResolutionInput& input,
        BattleRuntimeRandom& random) const;

    BattleDamageModifierState resolveDamageModifiers(
        const BattleEffectSources& attackerSources,
        const BattleHitResolutionInput& input,
        BattleRuntimeRandom& random) const;

    std::vector<BattleGameplayCommand> resolveStatusPayloads(
        const BattleEffectSources& attackerSources,
        const BattleHitResolutionInput& input,
        BattleRuntimeRandom& random) const;

    std::vector<BattleGameplayCommand> resolveResourcePayloads(
        const BattleEffectSources& attackerSources,
        const BattleHitResolutionInput& input) const;

    BattleHitFollowUpEffects resolveFollowUps(
        const BattleEffectSources& attackerSources,
        const BattleHitResolutionInput& input,
        BattleRuntimeRandom& random) const;
};
```

Cast-side 也需要同樣收斂，而不是只整理 hit resolver：

```cpp
class BattleCastEffectResolver
{
public:
    int resolveCooldownReductionPct(
        const BattleEffectSources& castSources) const;

    std::vector<BattleGameplayCommand> resolveCastCommitEffects(
        const BattleEffectSources& castSources,
        const BattleCastCommitContext& context,
        BattleRuntimeRandom& random) const;

    std::vector<BattleGameplayCommand> resolveCastFinishEffects(
        const BattleEffectSources& castSources,
        const BattleCastFinishContext& context,
        BattleRuntimeRandom& random) const;
};
```

每個 phase resolver 應同時負責 `Always` 與 triggered form。例如 poison 不應拆成一個 `Always` block 和另一個 OnHit block，而應由 `resolveStatusPayloads` 一次處理：

```text
PoisonDOT Always + PoisonDOT OnHit
    -> one BattleDamageRequest with poisonPct and poisonDurationFrames
    -> one log event
    -> activation recorded only for triggered effects that actually apply
```

### Combo-only API 作為 Adapter

`BattleComboTriggerSystem` 的 combo-facing API 可以保留，但行為主體應搬到 source-aware overload。

Preferred pattern：

```cpp
BattleProjectileBouncePrime BattleComboTriggerSystem::collectProjectileBouncePrime(
    const RoleComboState& state,
    const BattleProjectileBouncePrimeInput& input) const
{
    BattleEffectSources sources;
    sources.combo = { { BattleEffectSourceKind::Combo, BattleSkillSlot::None }, &state };
    return collectProjectileBouncePrime(sources, input);
}
```

需要 adapter 化的路徑：

- execute
- projectile bounce
- extra projectile count
- armor penetration
- selected hit statuses
- hit follow-ups
- cast commit effects
- cast finish effects
- paired value/value2 lookups

### Direct Combo Reads

混合來源 phase 中應避免直接讀：

```cpp
attackerCombo.sumAlways(EffectType::...)
attackerCombo.firstAlways(EffectType::...)
attackerCombo.effectIds(Trigger::OnHit, EffectType::...)
BattleComboTriggerSystem().collectTriggerEvents(attackerCombo, ...)
```

允許例外：

- defender-only passive effects where selected attacking skill is irrelevant
- adapter implementation
- frame-runtime combo-only logic where no selected skill exists
- battle-start unit-passive logic

第一階段不需要額外掃描工具；以 registry validation、code review、局部重構、focused tests 擋住即可。若後續仍頻繁回歸，再考慮更嚴格的自動檢查。

## Implementation Plan

### Step 1: Add `BattleEffectRuntimeRegistry`

新增 `src/battle/BattleEffectRuntimeRegistry.h/.cpp`。

內容：

- `BattleEffectRuntimePhase`
- `BattleEffectRuntimeRule`
- `isSelectedSkillMagicAllowed`
- `runtimePhaseFor`
- `isUltimateOnlyMagicEffect`

第一版 registry 應從目前 `magicEffectHasSelectedSkillScope` 的所有 accepted pair 遷移而來，並加入 cast-finish pair：

- `OnSkillTeamHeal` / `OnSkillTeamHealPct` + `Always`：`CastFinish`、`selectedSkillMagicAllowed = false`，只保留 combo payload 行為。
- `OnSkillTeamHeal` / `OnSkillTeamHealPct` + `OnUltimate`：`CastFinish`、`selectedSkillMagicAllowed = true`、`ultimateOnly = true`。

每個 pair 都要標記 runtime phase 與 selected-skill magic policy。不要讓 loader 用 `Always` 接受 magic 群療。

### Step 2: Make Loader Use Registry

在 magic effect parse/normalize 後，呼叫 battle-side registry 驗證。

這一步要刪除或替換 `ChessBattleEffects.cpp` 裡的 `magicEffectHasSelectedSkillScope`，避免 loader accepted scope 與 registry accepted scope 分裂成兩份清單。

Validation 應保持便宜：

- 不讀真實 config 以外的額外檔案。
- 不跑戰鬥模擬。
- 不掃描 source。
- 只對已解析出的 effects 查靜態表。

### Step 3: Preserve Ultimate-only Runtime Binding

維持 battle initialization 只把 magic effect state 綁到 ultimate skill。

也就是：

- `runtime.ultimate` 由 `actionPlan->ultimateSkill` 建立。
- 普通技能暫不掛 selected-skill magic state。
- `用途:` 不參與這個判斷。

若未來要放開普通技能，應改 registry policy 與 runtime binding，而不是用 YAML metadata 旁路。

### Step 4: Store Source-aware Cooldown Finish Ref

把 cooldown finish 的來源從 bool 提升成 source-aware ref。

也就是：

- cast 決策建立 cooldown 時，保存 `BattleSkillEffectRef`。
- 當前 release 只在 ultimate cast 時保存 `BattleSkillSlot::Ultimate`。
- 普通技能不保存 selected-skill magic ref。
- cooldown finish 後清掉 ref。
- 既有 `cooldownFromUltimate` 若仍需要供 combo trigger 判斷，可由 stored ref 推導，避免兩份狀態不同步。

同一步也要把 cast commit 的 pending mark 改成 source-aware：`OnSkillTeamHeal` pending 必須標在 combo 與 selected ultimate skill effect state 上，否則 finish resolver 即使拿到 skill source，也沒有 pending state 可消耗。

### Step 5: Extract Hit Phase Resolver

先從 hit runtime 拆出 phase helper，但範圍要完整覆蓋 hit 相關效果：

### Step 6: Extract Cast Planning / Commit / Finish Resolvers

整理 cast 相關效果，並把 selected-skill magic 與 combo-only payload 的 policy 留在 registry：

- `CDR`
- `PostSkillInvincFrames`
- `UltimateExtraProjectiles`
- `FlatShield`
- `CurrentHPPctBlast`
- `TeamMPRestore`
- `EnemyMpDamageAll`
- `LowestAllyHeal`
- `SpiralBleedProjectile`
- `MPRestore`
- `ControlImmunityFrames`

Cast finish resolver 只處理 cooldown 完成後的 source-aware `BattleEffectSources`。它不應自行推測「剛剛是不是絕招」，也不應直接讀 combo；來源必須由 cooldown finish ref 建好後傳入。`OnSkillTeamHeal` / `OnSkillTeamHealPct` 的 `Always` form 只能作為 consumed combo source 的 payload，不能作為 selected-skill magic payload。

### Step 7: Convert Combo APIs To Adapters

把混合來源會用到的 combo-only overload 轉成 adapter。

這一步的重點不是刪 API，而是確保行為主體只有 source-aware 版本。

### Step 8: Remove Known Duplicate Paths

逐一清掉目前幾個容易 drift 的複製路徑：

- execute resolution
- projectile bounce collection
- extra projectile collection
- cast-scoped invinc/shield/global effects
- cast-finish team heal effects
- hit status payloads
- paired runtime effects

### Step 9: Update Tests

測試不應依賴真實 `config/chess_magic_effects.yaml`。真實 config 的正確性由啟動載入驗證負責。

Unit tests 使用小型 synthetic definitions 測：

- registry 查詢
- loader 對缺 runtime phase 的 synthetic effect 失敗
- phase helper 對 combo-only / skill-only / combo+skill 都生效
- triggered activation count 寫回正確 source
- main projectile filtering
- paired value/value2 來自同一 effect instance
- ultimate-only binding 不會把 magic effect state 掛到普通技能
- magic `OnSkillTeamHeal` / `OnSkillTeamHealPct` 在 ultimate cooldown finish 時觸發
- magic cast-finish effect 的 activation count 寫回 selected ultimate skill effect state
- combo `Always` `OnSkillTeamHeal` / `OnSkillTeamHealPct` 在 pending 被消耗後仍會加總
- loader 拒絕 magic YAML 中 `Always` 的 `OnSkillTeamHeal` / `OnSkillTeamHealPct`
- normal skill cooldown finish 不會觸發 magic cast-finish effect

Integration tests 保留少量真實 hit/cast flow 測試，例如：

```text
selected ultimate skill + OnHit PoisonDOT
    -> accepted hit command has poisonPct
    -> poison duration comes from value2
    -> activation count is recorded on the skill effect state
```

## Current Risk Areas

這些地方最容易在半整併狀態下再次漏接：

- 同一效果同時有 `Always` 和 triggered form。
- `value` / `value2` 必須來自同一 effect instance。
- trigger 有 main-projectile 限制。
- follow-up projectile 必須攜帶 `skillEffectRef`。
- mixed-source hit/cast path 仍呼叫 combo-only helper。
- cast commit 分散在 `BattleCore.cpp` 多個區塊。
- cast finish source ref 必須在 cooldown 完成後清掉，避免下一次普通技能誤用上一個絕招來源。
